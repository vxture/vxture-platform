# Workplans

`workplans/` 用于存放阶段性、任务型、可清理的工作文档，重点记录任务推进过程中的临时计划、检查确认和后续待办。

它和 `docs/` 的边界不同：`docs/` 保存架构、设计、规范、部署基线和长期维护材料；`workplans/` 保存一次任务推进过程中的计划、跟踪、检查、人工确认和阶段复盘。

## 适合放入本目录

- 阶段计划：如 P5 / P7 / 发布推进计划
- 工作跟踪：任务列表、状态表、阻塞项、负责人记录
- 检查确认：上线前核对、PR 合入确认、晋级检查、人工操作记录
- 临时审计整改：某次审计或 review 的整改清单
- 发布执行记录：某次 beta / main 晋级过程记录
- 延后处理的小任务：当前不能立刻改，但已经明确需要排期的计划性工作

## 不适合放入本目录

- 架构设计、ADR、长期决策记录
- 编码规范、AI 工作规则、包边界说明
- 数据库 schema 设计、部署基线、环境拓扑
- 产品规格、权限模型、认证模型等长期有效文档
- 长期审计体系、规范性检查流程和规则库

## 建议命名

```text
YYYY-MM-DD-topic.md
phase-name-topic.md
release-YYYY-MM-DD.md
audit-remediation-YYYY-MM-DD.md
```

示例：

```text
2026-06-02-login-turnstile-fix.md
p7b-ruyin-cleanup.md
release-main-2026-06-02.md
```

## 生命周期

- `active`：正在推进或等待确认。
- `planned`：已确认需要做，但暂不进入当前代码改动。
- `done`：任务完成后可短期保留，便于追溯。
- `expired`：阶段完成、PR 合入、发布验证完成后，可批量删除。

清理前只需要确认没有未完成动作、未迁移的长期决策或仍需保留的证据链接。

## 和 docs 的关系

任务完成后，结果应沉淀到代码和 `docs/`，而不是长期依赖 `workplans/`。

- 代码改动进入对应包、配置或 workflow。
- 长期规则、架构结论、部署基线更新到 `docs/` 对应板块。
- 临时检查、执行记录、过程判断留在 `workplans/`，完成后可删除。
- 如果一份任务文档同时包含长期规则和过程记录，完成时必须拆分：长期内容进 `docs/`，过程内容留在 `workplans/` 或清理。

`docs/audit/` 属于长期审计规划、规范性检查和规则沉淀；某次任务中的临时 audit / review / checklist 归入 `workplans/`。

## 迁移原则

从 `docs/` 或根目录迁移文档前，先判断文档是否仍具备长期价值。

- 如果是长期设计或基线，留在 `docs/`。
- 如果是一次性任务推进记录，迁入 `workplans/`。
- 如果任务中沉淀出长期规则，先抽取规则更新到 `docs/`，再把任务记录迁入 `workplans/` 或清理。
