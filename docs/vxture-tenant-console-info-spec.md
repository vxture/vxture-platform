## 一、个人信息（Personal Profile）

用户在所有租户/工作区下共享的个人主页，与租户身份解耦。按"标识类 → 认证类 → 安全类 → 绑定类 → 归属类 → 数据类"的逻辑顺序组织，符合从"我是谁"到"我能做什么"的认知路径。

### 1.1 身份标识

- 头像
- 昵称（可自由修改，无频次限制）
- 用户名（唯一标识；**注册后可改，限制 30 天内最多变更 1 次**）
- 邮箱（唯一，含验证状态：未验证 / 已验证）
- 手机号（唯一，**主标识**，含验证状态：未验证 / 已验证）
- 个人简介
- 时区偏好
- 语言偏好

> 标识类字段的唯一性校验（用户名/邮箱/手机号）需在修改时做实时查重提示。

### 1.2 个人认证（Identity Verification）⭐改为独立详情页

> 非强制项。用户可自主发起，认证结果将被「个人租户」直接关联引用（见 3.5）。
> **页面形态：独立详情页**（非弹窗）。原因见文末「认证页面架构说明」——认证是状态机驱动的多阶段流程（草稿/审核中/失败/已认证），且需为后续接入三方认证服务预留"认证渠道选择"扩展位，弹窗信息密度和扩展性均不足。

**页面结构（与 3.4 企业认证页共用模板）：**

1. 顶部状态条：当前状态 + 关键时间节点时间线
2. 认证方式选择区（**当前阶段隐藏，仅系统内置一种认证方式**；为后续接入三方认证预留，届时升级为渠道选择卡片/Tab）
3. 表单详情区
4. 历史记录区（发起/审核/驳回记录）

- 认证状态（未认证 / 审核中 / 已认证 / 认证失败）
- 认证类型（个人身份证 / 护照 / 其他证件，按地区适配）
- 证件姓名（与证件一致，认证通过后只读）
- 证件号码（脱敏展示，如 `110***********1234`）
- 认证发起时间 / 审核完成时间
- 驳回原因展示（认证失败时）
- 重新提交认证入口

> ~~证件照片上传~~、~~认证有效期/定期复核~~ — 已确认不需要，本期按"一次认证、长期有效、无需上传证件影像"的轻量模式设计（具体认证方式由认证服务商在审核流程中处理，不在本页面采集影像类敏感信息）。

**待确认事项：**

- 是否对接第三方实名认证服务（如阿里云实人认证、腾讯云慧眼）？
- 是否区分国内/海外用户的认证流程差异？

### 1.3 三方认证（Third-party Verification）⭐规划占位

> 本期仅做 UI 占位展示，不开放实际接入，用于向用户预告能力规划。

- 认证渠道列表（如企业邮箱域名认证、LinkedIn 职业认证等 — 具体渠道待规划）
- 每项展示「即将上线」标签，禁用交互

### 1.4 三方账号绑定

> 锁定 4 个渠道，按接入状态分组展示。

| 渠道             | 状态                        |
| ---------------- | --------------------------- |

| 飞书（Feishu）   | 可绑定                      |
| 钉钉（DingTalk） | 可绑定                      |
| 微信（WeChat）   | 可绑定                      |

- 绑定 / 解绑操作
- 绑定状态展示（已绑定账号信息，如昵称/头像）
- SSO 登录可用性标识

### 1.5 账户安全

- 登录密码修改
- 双因素认证（2FA/MFA：TOTP / 短信 / 邮箱）
- 登录设备管理（当前会话列表，支持远程下线）
- 登录历史记录（IP、地点、时间、设备）
- Passkey / 安全密钥管理

### 1.6 我的工作区

- 当前角色与权限展示
- 快速切换入口
- 退出 Workspace / 租户操作

### 1.7 隐私与数据

- 个人数据导出（合规要求，如GDPR）
- 账号注销 / 删除申请
- 隐私协议与服务条款查看记录

## 三、租户信息（Tenant Profile）

### 3.1 概念结构

**租户类型：**
| 租户类型 | 说明 | Workspace 关系 |
|----------|------|----------------|
| 个人租户（Personal Tenant） | 用户个人主体创建 | 1个租户 = 1个默认 Workspace |
| 组织租户（Org Tenant） | 企业/团队主体 | 1个租户（Org）= 1个默认 Workspace（**当前阶段**），后续演进为 1:N |

> 当前阶段：无论个人 / 组织租户，均为 **1 租户 : 1 默认 Workspace**，组织租户的多 Workspace 能力为后续演进方向，本期 UI 上对 Workspace 整体做**弱化展示**（不作为一级导航重点，但数据模型与页面入口需预留）。

### 3.2 租户基本信息

- 租户名称
- 租户Logo
- 租户ID（只读）
- 租户类型（个人 / 组织，只读，创建后不可变更）
- 租户简介 / 描述
- 所属行业（组织租户）
- 企业规模 / 人数区间（组织租户）
- 官网地址（组织租户）
- 创建时间（只读）
- 租户状态（**正常 / 停用**，只读）
- 认证状态（只读摘要，详情见 3.4 独立页面）

> ~~当前套餐版本~~ — 已移除。套餐/订阅绑定在 **Workspace 级**，租户本身可能承载多个 WS、对应多个 Plan，租户页不展示单一套餐字段，改为展示「关联 Workspace 及各自套餐」的汇总列表（入口跳转账单板块）。

### 3.3 租户管理员 / 联系人

> 信息需完整，作为租户的官方对外联系窗口，与认证、合规、账单通知强相关。

- 管理员姓名
- 管理员职务/角色（如：法定代表人、财务负责人、技术对接人，可多选标签）
- 联系邮箱（建议与认证审核结果通知绑定）
- 联系电话
- 注册地址 / 办公地址
- 国家 / 地区
- 邮政编码
- 是否为账单接收人（标识，影响发票/账单通知投递对象）

### 3.4 租户认证（独立详情页）⭐重构为独立页面

> 个人认证（1.2）与企业认证均采用**独立详情页**，且共用同一套页面模板。认证类页面是状态机驱动的多阶段流程（草稿/审核中/失败/已认证）+ 历史时间线 + 文件类内容，弹窗的纵向空间和交互能力不足以承载；更关键的是，**后续接入三方认证服务（如阿里云实人认证、企业征信类API）时需要"渠道选择 + 跳转/回调"的页面结构**，独立页面有固定路由可承接三方回调并恢复状态，弹窗在跳转过程中容易丢失上下文。详细架构分析见文末附录。

**按租户类型分流：**

| 租户类型 | 认证主体                | 页面形态                                                  |
| -------- | ----------------------- | --------------------------------------------------------- |
| 个人租户 | 关联用户个人认证（1.2） | 只读摘要卡片 + "前往个人信息完成认证"跳转，**不单独建页** |
| 组织租户 | 企业认证                | **独立详情页**，完整表单                                  |

**组织租户 - 企业认证详情页字段：**

> 页面结构与 1.2 个人认证页共用模板：① 顶部状态条 ② 认证方式选择区（当前隐藏，预留三方渠道接入） ③ 表单详情区（下方字段） ④ 历史记录区

- 认证状态（未认证 / 审核中 / 已认证 / 认证失败）
- 企业全称（工商注册名）
- 统一社会信用代码 / 税号
- 营业执照上传
- 法定代表人姓名
- 驳回原因展示（认证失败时，含审核意见详情）
- 认证发起时间 / 审核完成时间
- 重新提交认证入口
- **变更记录区**：营业执照等关键信息一旦修改，**需重新走审核流程**，此处展示历史变更与对应审核记录

**前置条件（已决策）：**

- 组织租户发起企业认证前，**必须要求创建者/经办人本人已完成个人实名认证**（1.2），系统在企业认证入口处做前置校验，未完成个人认证则引导先完成。

### 3.5 认证状态归属层级（已决策）

- 认证状态挂在 **Tenant（租户）层**，作为租户的整体属性。
- 后续若个人租户支持多 Workspace，各 Workspace **继承** Tenant 层的认证状态，不在 Workspace 层重复展示/采集。

### 3.6 区域与本地化

- 默认时区
- 默认语言
- 默认货币单位

> 层级关系（已决策）：设置在 **租户层**，Workspace **继承**租户层配置，暂不支持 Workspace 级独立覆盖。

## 四、工作空间（Workspace）

> 整体定位：**当前阶段弱化展示**，不作为用户高频感知的一级概念（当前一个组织对应一个默认 Workspace），但数据结构与页面入口必须完整预留，为后续 Org → 多 WS 演进做准备。

### 4.1 Workspace 基本信息

- Workspace 名称
- Workspace 图标（**提供默认图标**，支持自定义上传）
- Workspace ID（只读）
- Workspace 描述
- 所属租户（只读，归属关系展示）
- 默认 Workspace 标识（组织内是否为默认进入的工作区）
- 创建时间 / 创建人（**只读，仅展示概要信息：创建人昵称**，不展示完整身份信息）
- Workspace 状态（正常 / 已归档）
- Workspace 切换 / 新建入口（组织租户专属；个人租户当前阶段隐藏或灰显）

### 4.2 Workspace 计费角色（已决策）

> 明确"租户管支付，Workspace 管消费"的分层模型，类似 AWS Organizations（账单合并到管理账户）/ 阿里云企业账号（统一结算）的思路。

| 层级               | 职责                                                                  |
| ------------------ | --------------------------------------------------------------------- |
| **租户（Tenant）** | 支付与承载主体：统一预付、统一结算、统一开具发票                      |
| **Workspace**      | 消费与订阅单位：各 WS 独立产生用量、独立订阅 Plan，费用明细按 WS 归集 |

- 租户层：账户余额、预付记录、统一发票、整体账单（汇总所有 WS 消费）
- Workspace 层：当前订阅的 Plan、用量明细、消费记录（明细数据上卷至租户层账单）

> 此模型确认后，3.2 中"套餐版本"字段从租户页移除、改为 Workspace 页展示，已在 3.2 中标注。

## 附录：认证页面架构说明（弹窗 vs 独立详情页）

### 为什么弹窗装不下

认证类页面是**状态机驱动的多阶段流程**，不是一次性表单提交：

- 多状态生命周期：未认证 → 表单填写中 → 审核中 → 已认证 / 认证失败 → 重新提交，每个状态展示内容不同
- 历史轨迹：发起时间、审核完成时间、驳回原因、变更记录，需要时间线展示，弹窗纵向空间不足
- 文件类内容：营业执照上传/预览/重新上传，弹窗内嵌文件预览交互体验差

### 接入三方认证后复杂度进一步上升

1. **跳转/回调流程需要稳定页面承载**：多数三方认证服务是"发起请求 → 跳转三方页面或拉起SDK → 完成验证 → 回调通知结果"的异步流程，回调需要落地到有固定路由、可刷新的页面。独立详情页（如 `/profile/verification`、`/tenant/verification`）可以直接承接回调并恢复状态；弹窗在用户跳转出去再返回时容易丢失上下文。
2. **多供应商并存需要"渠道选择层"**：一旦支持多个三方认证服务商，页面结构从"一个表单"变为"渠道列表 → 选中后展开对应表单/跳转流程"，弹窗内嵌套二级导航体验拥挤。
3. **跨渠道状态聚合与重试引导**：某渠道认证失败时，可能需要展示"是否尝试其他渠道"，这需要按渠道分组的版面（卡片/Tab），不是单一表单能承载的。

### 统一页面结构（个人认证 / 企业认证共用模板）

1. **顶部状态条**：当前认证状态 + 关键时间节点横向时间线
2. **认证方式选择区**：当前阶段隐藏（仅系统内置一种认证方式）；后续接入三方认证后升级为渠道选择卡片/Tab列表
3. **表单/详情区**：当前选中渠道对应字段
4. **历史记录区**：发起记录、审核记录、驳回原因、变更记录

**设计收益**：当前不接三方认证时页面不显空（步骤2隐藏即可），后续接入三方认证时无需重新设计页面骨架，只需在"认证方式选择区"扩展内容。
