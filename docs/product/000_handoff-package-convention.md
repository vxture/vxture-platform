# 产品交接包命名规范（000_handoff-package-convention）

> 版本：v1.0 · 日期：2026-07-10 · 适用：`docs/product/{product}/` 下每个对外产品的对接文档包
> 首个实例 = **arda**（`docs/product/arda/`）；后续产品照此模板建包。

---

## 0. 目的

每个"对接 vxture 平台"的产品（arda、以及后续 L2/L3 产品）在 `docs/product/{product}/` 下维护一个**编号交接包**。目标：

- 与 `docs/design/` 的 `{域}_{NNN}` 编号体系一致（**编号在文件名、版本在文档 header**）；
- 把"完整交接说明 / 接口契约 / 逐轮问答"三类角色**分文件、可独立演进**；
- 修正意见有明确的**沉淀去向**，不散在问答里丢失。

## 1. 命名：`{product}_{NNN}_{name}.md`

- `{product}` = 产品短名（如 `arda`）。
- `{NNN}` = 三位编号，按下表分带（预留扩展空间）。
- **版本不进文件名**——放文档 header（`版本：vX.Y`），避免升版引发重命名与链接漂移（与 `product_310` 等现例一致）。

## 2. 编号分带（模板）

| 编号                     | 文件                                                                                                                                                                  | 角色              | 版本 |
| ------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------- | ---- |
| `_000_definition`        | **产品定义**——域模型 / 范围 / 边界（产品自有依据）                                                                                                                    | 随产品            |
| `_100_handoff`           | **对接交接总纲**——完整描述对接项目（范围、架构、文档包清单、验收口径、边界纪律）。**所有修正最终沉淀于此**（与 200 分工：总纲 = 面向"怎么落地"，接口 = 面向"接什么"） | `vX.Y`（header）  |
| `_200_interface`         | **对接接口契约**——只表述接口（C1/C2/C3 端点、请求响应形状、值域、鉴权、事件），机器可实现。**版本与 100 锁步**                                                        | `vX.Y` = 100 版本 |
| `_300_integration-final` | **最终对接要求 + 决策留痕**——对接定型后，逐轮回函收敛成一份：最终产品侧义务 + 架构级决策（含被否决的旧方案，防回退）。逐轮 reply 是过程件，收敛后删除                 | `vX.Y`            |
| `_900_tracking`（可选）  | **落地 / 一致性追踪表**——产品已实现项 vs 平台验收状态                                                                                                                 | 活                |

> 对接进行中可临时用 `_3NN_reply-NN`（平台回函第 NN 轮，问答 + 该轮裁定）作过程件；**对接定型后一律收敛进 `_300_integration-final` 并删除逐轮 reply**——过程记录只保留"决策概要 + 被否决的旧方案（防回退）"，不留逐轮流水。

## 3. 版本与修正流

- **100 handoff 与 200 interface 同版本、同步升版**。接口的实质变更（新端点/字段/值域/鉴权方式）→ 两者一起升 minor/major。
- **reply 是过程，handoff/interface 是settled 状态**：某轮问答敲定的修正，**写回** 100 或 200 对应处，reply 里保留原始问答与裁定理由（可追溯）。reply 不作为长期契约来源。
- **值域/契约的最终权威始终在 `docs/design/` 的 `product_{NNN}` 与 `@vxture/shared`**；产品包是"面向该产品的合并视图"，与权威冲突以权威为准（各文 header 声明）。

## 4. 与 `docs/design/` 的关系

- `docs/design/product_{NNN}_*`（矩阵/共享/三通道契约/目录模型/命名/实施规划）= **平台侧架构与契约权威**，跨产品共用。
- `docs/product/{product}/{product}_{NNN}_*` = **该产品的对接落地包**，引用上述权威、不复制其语义。
- 平台侧"某产品对接实施规划"（如 `product_310_arda-integration`）留在 `docs/design/`（属平台架构族），产品包的 `_100_handoff` 指向它作上游。
