一致性审核

跨包发现的结构性问题与建议
⚠ admin-bff 多处直连 pg（绕过 service 层）
BillingRouter、InvoicesRouter、OrdersRouter、PaymentsRouter、TicketsRouter 均自持 Pool 直连数据库，未通过 service-billing / service-ticket。这绕过了分层约束，但目前这些 service 是 stub，属于务实选择。
建议：service-billing / service-ticket 接入真实 Prisma 仓库后，逐步将 BFF 切换为调用 service。
⚠ service-billing / service-ticket 依赖 core-api，但未使用
@vxture/core-api（HTTP 客户端）被列为依赖，但这两个 service 没有对外发 HTTP 请求的场景。这些依赖是历史遗留，增加了构建体积。
建议：从两个 stub service 的 package.json 中移除 core-api。
⚠ service-ticket 依赖 core-auth，但未在方法中使用
@vxture/core-auth 依赖声明存在，但 TicketService 的公共方法均不使用 JWT 验证功能（认证在 BFF middleware 层）。
建议：移除 core-auth 依赖，认证边界保持在 BFF 层。
ℹ 两个 mail 包的职责划分
core-mail（packages/core/mail）— 轻量 NestJS 全局模块，single send()，BFF 层 fire-and-forget 通知。无 Redis，no-op 安全。
service-mail（services/notification/mail）— 业务域服务，含 VerifyCodeService（Redis TTL）+ 批量发送，用于登录/注册验证码流程。
结论：划分合理，无重复。应保持各自职责，不合并。
ℹ core-mail 不依赖 core-config（设计决定）
core-mail 直接读 process.env.SMTP_* 而不通过 VxConfigService，原因是 core 包间禁止互相依赖。Consumer BFF 已通过 VxConfigModule 加载 .env，SMTP 变量在 MailService 构造时已可用。
结论：架构约束合理，无需修改。
▣ workers 包为空 stub
services/support/workers 只有 package.json + .gitkeep，无代码。M4（公告群发邮件）的批量异步逻辑计划在此实现（BullMQ）。
建议：M4 启动前设计队列接口，确定 Redis 连接配置后建包。
▣ ai-sdk 模块不完整
RAG / Embedding / Workflow 三个模块仅有类型定义，ModelRuntimeLLMClient 已实现。version 0.0.0 反映开发状态。
建议：以 LLM 模块为模板，RAG/Embedding 模块按需实现，勿提前扩张接口。
✓ Core 层包间零依赖约束满足
7 个 core 包中，core-api / auth / locale / tenant / utils 均只依赖 @vxture/shared；core-config 零内部依赖；core-mail 零内部依赖。没有 core 包引用其他 core 包。约束完全满足。
✓ Service 层包间零依赖约束满足
所有 service 包均未引用其他 service 包。跨域数据在 BFF 层通过 aggregator 组合。约束完全满足。
✓ 端口分配自洽
Platform BFFs：3011/3021/3031；Vela 三端：3120/3121/3122；Model Platform：3100。全部符合 3NNX 规则，已在 port-allocation.md 登记。
