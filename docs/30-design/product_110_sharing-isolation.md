## 1. 背景与目标

Vxture 为多租户 SaaS 平台,租户模型为 **org(tenant)→ 多个 workspace**,entitlement 粒度为 workspace × product(state × tier 双轴)。

L3 行业应用(Raven / Anlan / Forge / Xuanzhen)业务跨度大,各自建有专属数据库(**agent-db,一个产品一套,按 WS 隔离**);同时平台需要在 org 内实现数据、知识、能力的按需共享与复用。

本文档定义贯穿平台的统一共享与隔离模型,回答四个问题:

1. 什么是不可配置的硬边界,什么是可配置的软边界;
2. 三个 L2 域平台以什么统一模式托管与共享资产;
3. 能力如何供给 agent,Runa 在其中的角色是什么;
4. 共享如何表达、由谁授权、在哪里执行。

---

## 2. 核心设计原则

> **能力同构、数据异构、共享是策略而非结构。**

| 原则           | 含义                                                                                                                                                           |
| -------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **能力同构**   | 所有 L3 agent 消费同一套 L1/L2 能力面,无任何 agent 拥有私有能力;agent 间差异仅来自数据与场景编排。能力"存在性"同构,"可用性"由 entitlement 控制(feature-gating) |
| **数据异构**   | 数据按需下沉至 agent-db 或托管于 L2 域平台,归属始终清晰;托管程度由各对象域的数据结构收敛度决定(见 §4 托管水位线)                                               |
| **共享是策略** | 共享通过授权对象(SharingGrant / entitlement)表达,不产生数据副本;数据留在原地,可见性由策略动态计算,撤销即时生效、无残留                                         |
| **默认拒绝**   | org 内跨 WS / 跨 product 的访问默认关闭(default-deny),按需显式开放                                                                                             |
| **供给直连**   | L2 能力对 agent 的供给通道唯一为直连(L0 统一协议 + 各产品入口授权);不存在中心代理层                                                                            |

## 3. 两平面模型:硬隔离 + 软共享

### 3.1 技术平面(硬边界,不可配置)

- **org(tenant)是绝对隔离边界。** 所有存储(agent-db、Karda 索引、Arda 目录、Terra 租户空间数据、图谱实例)的数据键均携带 `org_id`;跨 org 的租户数据流动在架构上**不存在通路**——不是"权限不允许",而是"无此路径"。
- 唯一合法的跨 org 数据形态是 **P 级平台共有资产**(§4.2):平台供给、只读、经 entitlement 消费,不含任何租户数据。
- 该边界是 SaaS 安全底线,任何业务配置不得突破。已吸入平台数据架构铁律表(`data_platform_100` §2.2.3)。

### 3.2 业务平面(软边界,策略驱动)

- org 内部以 **WS 为共享单元**,通过权限配置实现按需开放。
- 隔离不靠数据搬运或复制实现,靠策略层的授权对象实现。
- 推论:**共享不产生数据副本**——WS-A 向 WS-B 开放某资产,仅在策略层登记一条 grant,访问时按调用方身份动态求交;撤销即时生效,审计只需策略变更记录。

## 4. L2 域平台统一原型(Archetype)

**L2 只有一种模式,三个产品(Karda / Terra / Arda)是同一模式在不同参数下的实例。** 这是本模型的核心公理。

### 4.1 原型结构

```
L2 域平台 = 能力层 + 分级资产层 + 授权层

能力层:域内的加工/计算/检索/治理能力
        以一方工具形式按 L0 统一协议直连暴露,entitlement 控制可用性
资产层:P 级 — 平台共有资产(跨 tenant、只读、产品级供给、entitlement 消费)
        T 级 — 租户资产(org/WS 隔离、SharingGrant 授权共享)
        A 级 — agent 专属数据(托管在域平台内 或 驻留 agent-db)
授权层:跨 tenant 走 entitlement;org 内走 SharingGrant;两套机制不混用
```

### 4.2 P 级平台共有资产(平台级通用机制)

P 级资产是 L2 通用概念,在 L0/本模型统一定义一次,三个 L2(及 Runa 的平台技能)复用。

**范围界定(v1.0 收窄,owner 拍板)**:P 级资产 = **行业公开数据 + 三方公开服务数据**(如高德地图、Google 影像、公开法规/标准库、行政区划等)。**敏感数据不进平台**——含个人信息、涉密、受管制(如测绘管制)的数据不得作为 P 级供给;需要此类数据的场景走租户自有(T/A 级)通道并由租户承担合规主体责任。

**供给形态二分**:

| 形态                               | 定义                                                            | 数据落点                                          | 典型                                                |
| ---------------------------------- | --------------------------------------------------------------- | ------------------------------------------------- | --------------------------------------------------- |
| **自有资产(asset)**                | 平台自持、版本化托管的行业公开数据                              | 平台侧(所属 L2 的产品数据面),版本化管理           | 法规/标准/通用 SOP 库、行政区划、行业代码、通用白膜 |
| **代理持牌服务(brokered service)** | 三方公开服务的代理接入,平台售卖的是**接入与额度**而非数据再分发 | 数据不落平台(或仅协议允许的缓存);许可主体在服务商 | 高德地图瓦片、Google/商业遥感影像服务、天地图       |

> 地图/遥感类受《测绘法》等监管,平台自持再分发不可行,一律按 brokered service 建模;许可与合规元数据指向服务商条款。

**通用机制**:

- **供给方**:Vxture 平台(产品级支持),跨 tenant 只读,版本化管理;
- **消费方式**:经 entitlement 消费(可作为独立 SKU 或 tier 权益售卖),**不走 grant**(grant 是 org 内机制);
- **元数据要求**:含**来源与许可合规元数据**字段——数据来源(provenance)、许可条款与证据、再分发范围、地域合规约束、版本;
- **来源审计(v1.0 强化,owner 拍板)**:P 级资产全生命周期审计——①**准入登记**(来源、许可证据、再分发范围、责任 operator,走上架准入流程,对齐 `product.launch_checklist` 模式);②**版本变更审计**;③**分发消费审计**(哪个 org/WS 经何 entitlement 消费、用量);④**许可到期/变更复审**。审计记录入平台中央审计(`support.audit_log`);
- **商业意义**:平台自有的知识/数据资产成为独立收入线,是纯能力平台不具备的。

### 4.3 托管水位线:三个 L2 的主变量

三个产品之间的主变量是 **A 级数据的托管水位线**——即 agent 专属数据有多大比例可被域平台托管。水位线由客观属性决定:**该对象域的数据结构收敛度**。另有一个正交的次变量:**运行时托管**(域平台是否托管持续运行的实例态)。

| 维度                    | Karda                                  | Terra                                  | Arda                                        |
| ----------------------- | -------------------------------------- | -------------------------------------- | ------------------------------------------- |
| 对象域                  | 非结构化知识                           | 时空/物理世界                          | 结构化业务数据                              |
| 数据结构收敛度          | 高(加工后表示统一:分块/向量/图谱/索引) | 分裂(底座标准统一、业务数据发散)       | 低(schema 随业务各异)                       |
| **托管水位线**          | **全量托管**                           | **托管标准化底座**                     | **托管通用件,其余汇聚**                     |
| **运行时托管**(次变量)  | 索引即运行时(检索不能脱离索引)         | 孪生实例运行态由 Terra 托管            | 无(目录与服务化,无常驻实例)                 |
| P 级资产示例            | 平台知识库(法规、标准、通用 SOP)=asset | 地图、遥感影像=brokered;通用白膜=asset | 通用参考/主数据(行政区划、行业代码等)=asset |
| A 级数据驻留            | Karda 内,以授权逻辑区分                | agent-db(保留比例按业务复杂度自定)     | agent-db 为主                               |
| agent 是否需自建域内 db | **基本不需要**                         | 需要,规模按业务定                      | 需要(即 agent-db 本身)                      |
| 隔离手段                | 授权逻辑 + 索引命名空间(§10#4 分档)    | 平台底座跨 tenant / 租户数据隔离       | 库级物理隔离 + 目录授权                     |

## 5. 三个 L2 的参数化定位

### 5.1 Karda:知识能力域 + 全量知识资产托管

**定位**:企业知识能力域——知识加工、检索与治理能力,托管平台/组织/agent 三级知识库,整体被 agent 集成应用。

```
Karda(L2 知识能力域)
├── 能力层(Knowledge Engine)
│     解析/分块/向量化/图谱抽取/混合检索/重排/知识治理(验证·时效)
│     按 L0 协议直连暴露,entitlement 控制可用性
│
└── 资产层(Knowledge Bases,对应 P-T-A 三级)
      P:平台知识库 — 平台供给,跨 org 只读,entitlement 售卖
      T:org 知识库 — org 管理员建设,org 内按配置可见
      A:agent 知识库 — agent 属主(org × ws × product 键),默认私有,
         经 SharingGrant 授权给其他 WS/product
```

要点:

- **所有权与托管分离**:知识库由 agent / org 拥有(ownership),由 Karda 托管(hosting)——索引态资产(向量索引、图谱实例、分块存储)落在 Karda 基础设施内,检索能力无法脱离索引运行;
- **知识流动由属主发起**:属主建库 → 绑定 Arda DataSource 或直接上传 → Karda 加工;不存在"平台自动拉通汇聚"。agent 运行沉淀的知识(如客服 QA 对)由 agent 用 Karda 能力写入自有库,再按 grant 开放,无需单独的"回流"机制;
- **派生边 day-one 建模(v1.0 硬约束)**:Karda 在 **ingestion 时点**为每个派生资产(分块/向量/图谱条目)记录派生边——`派生资产 → 来源 DataSource/文档 → 授权依据(grant_id 或自有)`。这是级联撤销(§8.6)的前提:派生关系若事后补记,已建索引无派生记录,撤销无从级联。写入 Karda 产品定义硬约束;
- **跨库统一检索为一等能力**:调用方给身份,Karda 在其全部可见范围(自有库 ∪ 被授权库 ∪ org 库 ∪ 已订阅平台库)上做联合检索 + 统一重排——保留 org 级"什么都能问"的体验,语义为"调用方可见范围的并集"而非"平台拉通一切"。可见范围求值复用 §8.5 物化可见集,联合检索扇出与策略求值共用一份物化;
- **授权即隔离**:Karda 无 agent-db 物理兜底,agent 知识间的边界完全由授权逻辑 + 索引命名空间成立,§10 两条硬约束适用;
- **例外条款**:极端敏感或有驻留合规要求的知识,允许 agent 侧申请独立索引域(silo-on-demand 通路),例外审批而非默认选项。

### 5.2 Terra:物理世界能力域 + 标准化底座托管

**定位**:数字孪生 + 基础地理 + 物联感知的横向能力底座,附带标准化空间底座数据的跨租户供给。

- 能力层:时空计算、孪生建模与运行、感知接入等,直连暴露;
- P 级:地图、遥感影像等三方公开服务一律按 **brokered service** 形态供给(§4.2,许可主体在持牌服务商,平台卖接入与额度);通用白膜等结构统一(OGC 等标准背书)且许可清晰的底座数据按 asset 形态供给;两形态均须携带来源与许可合规元数据并受来源审计;
- T/A 级:租户自有空间数据严格隔离;业务层数据(物联感知流、孪生业务实体)结构因业务而异,留在 agent-db,保留比例由各 agent 按业务复杂度自行确定;孪生实例的运行态由 Terra 托管(运行时托管,§4.3 次变量)。

### 5.3 Arda:数据服务平台,通用件托管 + 汇聚共享

**定位**:通用结构化数据 + 数据汇聚共享。agent-db 是数据的 System of Record,Arda 是数据的 System of Access。

- 能力层:数据目录、数据服务化输出、连接与同步调度,直连暴露;
- P 级:跨业务复用的通用参考/主数据(v1 清单与准入流程归 Arda 产品定义,见 §12 决策#4;准入判据对齐 Rule of Two 精神——≥2 个 agent 需要且无租户属性才进 P 级);
- T/A 级:业务数据驻留 agent-db(一产品一套、WS 隔离),Arda 目录层记录 `(org, ws, product, datasource)` 四元组;跨 agent 数据共享经 Arda 授权语义显式声明,默认不共享。

### 5.4 连接器归属:连接 = Arda,理解 = Karda

- **连接器注册与目录归 Arda**。连接器是 DataSource 的类型化实现:① 内部 agent-db(首要类型);② 外部结构化源;③ 外部非结构化源(Notion / 飞书 / SharePoint / 对象存储等)。"平台知道哪里有数据"的唯一登记处是 Arda;
- **非结构化内容的消费管线归 Karda**。Arda 负责连接、鉴权、同步调度;内容经通道交付 Karda 完成解析、分块、向量化、图谱抽取(并按 §5.1 记录派生边)。管线由**属主拉动**(建库时绑定 DataSource),而非平台推动;
- Karda 对 Arda 的依赖为实现层依赖,不上升为矩阵结构概念。

## 6. Runa:技能与专用模型资产平台

### 6.1 定位

**Runa 是服务 agent 的技能库与小模型整合层**(参照 Anthropic Skills / OpenClaw 技能库形态):将 Prompt、专用小模型与工具引用组合打包为可分发、可版本化、可授权的技能单元(Skill)。**Runa 是纯控制面/资产面产品,自身不持有数据面运行时。**(精确判定见 §6.7。)

### 6.2 Skill 构成

```
Skill = 指令/Prompt 模板 + 编排逻辑与少样本示例
      + 专用模型绑定(分类器/抽取器/NER/OCR/rerank 微调版等,经 Atlas 调用)
      + 工具引用(引用 L1/L2 一方工具,如 Karda 检索、Terra 地理编码)
      + 元数据(版本、评测基线、entitlement/grant 声明)
```

技能资产同样遵循 P-T-A 三级供给:平台技能(跨 tenant,entitlement 消费)/ org 技能(租户自建,grant 共享)/ agent 私有技能——Runa 亦为 §4 统一原型的实例,其"资产"是技能而非数据。

### 6.3 Runa 的职责(仅四项,全部为控制面)

1. **技能定义与生命周期**:创建、版本、评测基线(eval)、发布与废弃。**eval 的执行不在 Runa**:评测集与基线定义、结果记录持有在 Runa;评测执行按 §6.6 构件就位模型走——推理在 Atlas、编排执行在 L0 沙箱(评测作业),结果回写 Runa 目录;
2. **技能目录与分发**:agent 按 entitlement/grant 发现并装载可用技能;分发为**拉取式**(agent 启动或按需拉取技能定义与签名构件),非推送式常驻通道;
3. **专用小模型的能力化**:任务层包装与目录登记(如"抽取模型 + Prompt 套件 = 合同要素抽取技能");**模型接入、推理与计量本身归 Atlas**,推理流量不经过 Runa;
4. **跨域组合**:将多个域的原子能力组合为任务能力(如"合规审查技能 = Karda 法规检索 + 条款抽取小模型 + 比对 Prompt")——单一 L2 做不了、agent 各自做会重复,是 Runa 的价值锚点。

### 6.4 Runa 不是什么(负面清单)

- 不是 L2 的 API 网关:L2 原子能力不经 Runa 转发;
- 不是模型平台:模型接入/路由/配额/推理宿主归 Atlas(含大模型与专用小模型);
- 不是 agent 的工作流引擎:agent 内部业务编排归 agent 自身,Runa 只提供可被编排的"件";
- 不是唯一授权点:L2 直连能力的授权在 L2 自己的入口执行,Runa 只管技能资产自身的授权;
- **不出现在任何调用链路上**(无中心瓶颈,无 ESB 反模式)。

### 6.5 技能准入判据(Rule of Two)

一段任务逻辑仅一个 agent 使用 → 留在该 agent 内部;被两个及以上 agent 复用,或需作为平台技能 SKU 售卖 → 沉淀为 Runa 技能。可操作化为三问:是否跨域组合?是否绑定专用模型?是否包含值得复用的任务逻辑?——三者有其一进 Runa,否则不进。

### 6.6 技能执行位置模型:构件分布式就位

技能执行不是"在 Runa 跑还是在 agent 跑"的二选一;Skill 的每类构件按各自天然归属就位执行:

| Skill 构件                             | 执行位置                                                                  | 行业对应                                                  |
| -------------------------------------- | ------------------------------------------------------------------------- | --------------------------------------------------------- |
| 指令/Prompt/编排逻辑(声明性部分)       | **agent 运行时**:技能定义分发装载进 agent 上下文,由 agent 的 LLM 循环消费 | Anthropic Skills、OpenClaw(技能库分发定义,agent 自行执行) |
| 工具引用                               | **提供方服务**:以调用方 agent 身份直连 L1/L2 入口执行                     | MCP(协议统一,provider 各跑各的 server)                    |
| 专用小模型推理                         | **Atlas 模型端点**:小模型注册为 Atlas 管理的专用模型,与大模型同路         | Bedrock 自定义模型 / SageMaker Endpoint                   |
| 重型非模型计算(OCR 流水线、代码执行等) | **L0 共享沙箱执行器**(平台级基础设施,非 Runa 产品能力)                    | 各家 code interpreter / sandbox 模式                      |

**小模型宿主为 Atlas 的三点理由**(已固化):① Atlas 的产品定义即模型接入/路由/配额/用量治理,专用小模型只是模型子类,不建第二个模型平台;② 租户计量口径必须唯一,推理量不过 Atlas 则计费体系分裂;③ GPU 资源池、弹性伸缩、模型版本管理等重资产运维能力平台只建一套。

边界情况:技能需"作为单个工具被一次性原子调用"时,由 L0 沙箱按技能定义解释执行,或由技能的主要提供域承载——Runa 仍不在调用链路上。

推论:**不存在"Runa 托管执行器"选项**;Runa 无推理与执行负载,为纯元数据服务,SaaS 成本模型极轻。

### 6.7 「无运行时」的精确判定(v1.0 新增,控制面/运行时职责表)

逐生命周期环节判定 Runa 的职责边界,防止"无运行时"被任何环节撕开口子:

| 生命周期环节 | Runa 做什么(控制面)                              | 执行/负载发生在哪                                                            |
| ------------ | ------------------------------------------------ | ---------------------------------------------------------------------------- |
| 创作 / 版本  | 技能定义 CRUD、版本树、构件签名                  | —(纯元数据)                                                                  |
| 评测(eval)   | 持有评测集定义、基线与历史结果                   | 推理 = Atlas;评测作业编排执行 = L0 沙箱;结果回写 Runa                        |
| 发布 / 分发  | 目录登记、entitlement/grant 声明、拉取式分发接口 | agent 运行时拉取并装载                                                       |
| 执行         | **无**                                           | 声明部分 = agent LLM 循环;工具引用 = L2 直连;小模型 = Atlas;重计算 = L0 沙箱 |
| 计量         | **零计量路径**(技能"使用数"仅目录统计,不作计费)  | 推理量计给 Atlas → consume;工具调用计给提供方 L2 → consume                   |
| 废弃 / 撤销  | 目录状态变更(deprecated/revoked)                 | agent 下次拉取生效;**不热杀运行中会话**(装载态技能随会话自然结束)            |

版本一致性约定:agent 装载技能采用**版本 pinning**(锁定至具体版本,升级为显式动作),避免技能定义静默漂移改变 agent 行为;技能定义构件带签名,agent 运行时装载前验签。

## 7. 能力供给:唯一直连通道

### 7.1 L0 工具协议规范(已固化)

能力以什么统一接口形态暴露(工具 schema、调用规范、鉴权与身份透传约定)是 **L0 级平台技术标准**(MCP 风格):由 L0 定义规范文档,各产品(L1/L2)负责实现。协议统一 ≠ 物理网关——不设中心代理节点。

> 规范文档为独立待建项(族内编号 `product_210_tool-protocol.md`,登记于 [`product_200_integration.md`](./product_200_integration.md) §5):须覆盖工具 schema 约定、**S2S 鉴权与调用方身份透传**(token exchange / on-behalf-of,现有接入标准只覆盖用户级 OIDC)、grant ∧ entitlement 求值时点、审计与计量归属。这是本模型落地的最大工程前置。

### 7.2 唯一供给通道:直连(已固化)

**L2 的能力供给通道只有一条——直连。Runa 不构成第二条供给通道。**

- L2 为独立产品,须有产品内的完整体现:自有 Console 体验、自有 API/工具面、自有入口授权求值;能力对 agent 的暴露是产品完整性的一部分;
- 若设"经 Runa 访问 L2"的第二通道,将产生:同一能力双接口面(版本演进双处同步)、授权双重求值(语义可能不一致)、L2 产品完整性受损(用量与体验数据旁落)——**必要性分析结论:不必要,且有害,明确不做**;
- **Runa 位于消费侧而非供给侧**:它供给的是"引用了 L2 能力的技能";技能运行时对 L2 的每次调用,仍以调用方 agent 身份走唯一直连通道,权限链路、计量归属、版本契约全部落在 L2 自己的入口。

### 7.3 供给结构总图

```
L2(Karda/Terra/Arda) ══唯一直连通道(L0 协议 + 各自入口 grant∧entitlement 求值)══→ agent
                                     ↑ 工具引用(以 agent 身份调用)
Runa(技能资产:定义/版本/评测/分发,纯控制面) ──分发──→ agent 运行时装载
Atlas ──模型推理(大模型 + 技能绑定的专用小模型,统一计量)──→ agent / 技能执行
Ontos(实体/关系/语义 Schema 定义) ──被 L2 消费(Schema 归 Ontos,实例归各 L2)
L0 vxture:org/WS/entitlement/计量计费/工具协议规范/共享沙箱/sharing 策略 SoT 贯穿全部
Terra P 级底座(地图/影像=brokered,白膜=asset) ──跨 tenant 只读──→ 各 org(entitlement 消费)
```

## 8. 共享策略模型(SharingGrant)

### 8.1 最小策略模型

```
SharingGrant {
  resource:    数据集 / 知识库 / 技能(归属键 = org + ws + product,资产 id 为跨面 loose 引用)
  grantee:     org 内的某 WS | 某 product | org-all
  scope:       按 resource 类型参数化(§8.2),区分"可召回"与"可应用"两级敏感度
  status:      active / revoked
  expires_at:  可选到期时间(临时协作/外审场景;到期即失效,与撤销是两条独立轴)
}
```

### 8.2 scope 按资产类型参数化(v1.0 细化)

| resource 类型(资产面) | scope 值域           | 语义                                                                                                                    |
| --------------------- | -------------------- | ----------------------------------------------------------------------------------------------------------------------- |
| 数据集(Arda)          | `read`               | 可经 Arda 服务面查询/读取(不含写;写永远只属属主)                                                                        |
| 知识库(Karda)         | `retrieve` < `apply` | `retrieve` = 允许被检索召回、生成时**必须带出处引用**;`apply` = 允许用于生成与应用(无引用生成)。`apply` 蕴含 `retrieve` |
| 技能(Runa)            | `use`                | 可被 grantee 发现、装载并执行                                                                                           |

### 8.3 命中谓词与联合求值(v1.0 写死,防实现分叉)

```
grant 命中(caller, resource) :=
      status = 'active'
  ∧  (expires_at IS NULL ∨ now() < expires_at)
  ∧  (   grantee = org-all
      ∨ (grantee = WS      ∧ caller.workspace_id = grantee.workspace_id)
      ∨ (grantee = product ∧ caller.product_id   = grantee.product_id) )

可访问(caller, resource) :=
      属主访问(caller.ws = resource.ws ∧ caller.product = resource.product)   -- 自有资产无需 grant
   ∨ (∃ 命中 grant ∧ entitlement_active(caller.workspace_id, 执行点产品))
```

- **grantee = product 的精确语义**:本 org 内、经由该 product 访问的**任意 WS 实例**(与 entitlement 的 workspace × product 双轴对齐,联合求值中的 entitlement 项继续约束具体 WS);
- **联合求值的 entitlement 操作数 = 执行点产品**(承载/服务该资产的 L1/L2,如知识库之于 Karda):调用方 WS 须持有执行点产品的有效 entitlement;**资产归属产品的订阅不作要求**(共享即授权),属主性由归属键保证;
- **多 grant 合成**:同一 (resource, caller) 命中多条 grant 时,scope **就高合成**(各条先独立判定 status/expires,有效者取最强 scope);不同 resource 之间互不影响;
- **撤销与重授**:撤销 = status 置 `revoked`(行保留,审计需要);重新授权 = 新建 grant 行,不复活旧行;
- **T 级 org 库的可见性同样以 grant 表达**(org-all 预设 grant),三级可见性(org 共享 / WS 共享 / agent 专属)是 grant 预设形式,不是数据的静态属性;
- **P 级资产不走 grant**:跨 tenant 的平台资产供给走 entitlement(§4.2),grant 仅为 org 内机制。

### 8.4 设计要点(承 v0.2,继续有效)

1. **默认拒绝(default-deny)**:无 grant 即不可见;
2. **策略执行点在各资产面入口**:Arda(数据)、Karda(知识)在各自服务入口执行 grant 求值;Runa 仅对技能资产自身执行;**权限过滤在召回层强制执行,不做生成后裁剪**;
3. **默认值差异**:数据默认私有、显式共享;知识默认按作用域声明——数据敏感度高于知识;
4. **可信度元数据随共享透出**(可选属性):跨 WS 消费时,资源的验证/时效状态作为元数据透出(非新 scope),服务受监管场景。

### 8.5 策略 SoT 与物化(v1.0 落位,owner 拍板)

**SoT = 平台控制面新增 `sharing` 域(schema)。** 理由:grant 横贯数据/知识/技能三类资产、跨 L2 产品,任何单个 L2 都不是自然属主;联合求值 grant ∧ entitlement 的两个操作数同源(控制面)才能一处求值、一处物化、一处失效;org 管理员全局审计与一键回收天然是平台 Console/Admin 界面;审计走现成 `support.audit_log`。

```
sharing.grants                    -- SoT
  id / tenant_id(FK,org 内机制硬约束)
  resource_type(dataset|knowledge_base|skill)
  resource_product_id(FK) / resource_workspace_id(FK)
  resource_ref                    -- 业务面资产 id,跨面 loose 引用(铁律一边界#1)
  grantee_type(workspace|product|org_all) + grantee_workspace_id? / grantee_product_id?
  scope / status(active|revoked) / expires_at?
  created_by / revoked_at / revoked_by

sharing.visible_set_current       -- 物化可见集(非 SoT,对齐 entitlement_current 模式)
  (grantee 键, resource 键, scope) UPSERT + 短 TTL
```

- **求值执行点不变**(仍在各 L2 入口,召回层强制);控制面提供解析 API + 按 grantee 预展开的物化可见集 + **invalidate 推送**(grant 变更秒级生效,复用 entitlement invalidate 通道,见对接标准 C3);
- 字段级 DDL 归后续 `data_sharing_100/200` 设计线(命名遵循 `data_{domain}_{NNN}` 规则);
- Karda 跨库统一检索(§5.1)与策略求值共用同一份物化可见集,检索扇出不重复求值。

### 8.6 级联撤销(lineage-aware revocation,v1.0 机制化)

覆盖"数据已撤销、派生知识索引仍可召回"风险(§10#2):

1. **派生边在 ingestion 时点记录**(§5.1 硬约束):`派生资产 → 来源 DataSource/文档 → 授权依据(grant_id | 自有)`,存于 Karda/Arda 业务面元数据;
2. **撤销事件广播**:grant 撤销/到期、DataSource 解绑在控制面产生事件,经 invalidate 通道推送相关 L2;
3. **re-scope 而非 re-index**:L2 按派生边将受影响的派生资产移出可见集(索引条目失效标记),不重建索引;召回层过滤保证即时不可见;
4. 审计:撤销事件与受影响资产计数写审计,供合规追溯。

### 8.7 管理权模型(v1.0 已固化,原 §8.3 建议转正)

**WS 发起 + org 审计回收**:grant 由资源所属 WS 的管理员发起;org 管理员保留全局审计与**一键回收**权;资源属主 WS 可随时撤销自己发出的 grant。全部 grant 创建/撤销/到期事件写 `support.audit_log`(actor / resource / grantee / scope)。

## 9. 行业对标结论(摘要)

- **隔离**:org 硬隔离 + 共享 Schema 键隔离对齐 Salesforce 经典多租户;org → 多 WS 结构对齐 Slack Enterprise Grid;相对 Glean 单租户部署更轻,以 per-tenant 加密(BYOK)+ 数据驻留选项(高阶 tier)+ silo-on-demand 预留通路作为补偿演进;
- **共享**:"共享不复制"对齐 Snowflake Secure Data Sharing;"策略执行点收敛于资产面"对齐 Databricks Unity Catalog;"召回层执行权限"对齐 Glean permission-aware retrieval;grant 到期语义对齐 Databricks sharing expiration;**grant 语义统一横贯数据/知识/技能是超出行业常规的部分**——这正是策略 SoT 必须单一(§8.5)的原因;
- **能力**:"能力同构、配置分化"对齐 Salesforce 单平台内核模式;"底座供养 agent"对齐 Glean / Agentforce;与 Dify/Coze 的"agent 私有为默认"取相反默认值,以避免知识碎片化;Runa 的分布式执行模型对齐 MCP + Anthropic Skills 生态,规避 ESB 反模式;
- **叙事差异点**:比 Glean 多出 Ontos 语义面与 Arda 数据面的显式建模,且以 L2 统一原型(P-T-A + 托管水位线)将其结构化。

## 13. 已固化项(本文档确认,无需再议)

- **L2 统一原型**:能力层 + P-T-A 三级资产层 + 授权层;三个 L2 为同一模式的参数化实例,主变量为托管水位线(由数据结构收敛度决定),次变量为运行时托管;
- **托管水位线取值**:Karda 全量托管(agent 基本不需自建知识 db)/ Terra 托管标准化底座(业务数据留 agent-db,比例自定)/ Arda 托管通用件 + 汇聚共享;
- **P 级资产**:范围 = 行业公开数据 + 三方公开服务数据(敏感数据不进平台);形态二分 asset / brokered service;经 entitlement 消费不走 grant;全生命周期来源审计;
- **连接 = Arda,理解 = Karda**;知识管线由属主拉动;知识沉淀直接写入自有库,无独立"回流"机制;**派生边 ingestion 时点记录**(级联撤销前提);
- **跨库统一检索**为 Karda 一等能力,语义为"调用方可见范围的并集",与策略求值共用物化可见集;
- **工具协议规范归 L0**:平台技术标准,各产品实现,不设物理网关;含 S2S 身份透传(规范文档待建,最大工程前置);
- **L2 供给通道唯一为直连**;Runa 位于消费侧,不构成第二通道;
- **Runa = 技能与专用模型资产平台**,纯控制面,无运行时(§6.7 判定表);技能准入遵循 Rule of Two;技能装载版本 pinning + 构件签名;
- **技能执行按构件分布式就位**:声明部分在 agent 运行时、工具引用直连提供方、小模型推理在 Atlas、重计算在 L0 沙箱;eval 执行同此(Runa 只存定义与结果);
- **小模型宿主为 Atlas**(计量口径唯一、模型平台唯一、重资产运维唯一);
- **共享不产生数据副本**;权限过滤在召回层强制执行;org 为绝对隔离边界,P 级资产为唯一合法跨 tenant 形态;
- **SharingGrant**:SoT 在平台控制面 `sharing` 域;命中谓词与联合求值按 §8.3;scope 按资产类型参数化;多 grant 就高合成;撤销保留行、重授新行;管理权 = WS 发起 + org 审计回收;
- 沿用既定:entitlement 粒度 workspace × product(state × tier);L0 vxture 不作为 product code;Ruyin(client 端)/umbra 不进 entitlement 新引擎(umbra 保持现状订阅);agent-db 一产品一套、按 WS 隔离。
