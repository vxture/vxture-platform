# CI/CD 部署提效 / 去冗余 Playbook

**用途**：交给 AI（或工程师）优化任意仓库的 CI/CD —— 去冗余触发、消减重复 job、最小化部署、补齐覆盖缺口。
**来源**：从 vxture 实战萃取（#150–#157 批次），通用、可迁移。
**版本**：1.0.0 ｜ **更新**：2026-06-09

> 本文是"任务说明书"：直接作为 AI coding 的指令/上下文使用。每节带可执行判定与验收口径。

---

## 1. 适用判定（何时做 / 不该做）

**该做**：workflow 数量多、存在 `workflow_run` 链、同一 commit 被多处重复校验、每次发布全量重建所有镜像/容器、部署耗时随服务数线性增长。

**不该做 / 暂缓**：仓库无分支保护与 required check（先补齐闸门再提效，否则去重会留缺口）；团队对发布流程无共识；缺少可观测的运行日志（无法实测收益，先补度量）。

---

## 2. 前置盘点清单（动手前必须先列清）

1. **全量梳理**：列出所有 workflow 的 **触发条件 / job / `needs` 依赖**，画出 `提交 → 校验 → 构建 → 部署` 完整数据流。
2. **逐项找四类问题**：重复触发、重复 job、过重部署、覆盖缺口。每项给**量化影响**（省多少分钟 / 少构建几个产物 / 少重启几个容器）。
3. **度量基线（关键，验收前提）**：动手前记录基线指标——
   - 每次发布触发的 **CI 运行数**
   - 每次发布的 **镜像构建数**
   - 每次部署 **recreate 的容器数**
   - **端到端耗时**（提交→生产就绪）
     > 没有基线，"提效收益"无法量化验收。收益必须用**优化后真实运行日志**对比基线，不靠推断。

---

## 3. 优化手法（编号，可直接套用）

### A. 去冗余触发

同一 commit 不重复跑同一套校验。典型：快进（FF）发布后，目标分支又跑一遍与源分支**完全相同 sha** 的 CI = 纯浪费。
**改法**：下游（构建）直接由 `push` 触发；前提该 sha 已被上游验证过（发布流程已校验 + 分支保护 required check 已覆盖）。

### B. 按变更构建（path → artifact 映射）

把"变更文件 → 受影响产物/镜像"映射出来，**只构建变更的**；未变更的复制旧产物引用（如 `docker buildx imagetools create` 把旧 tag 重打到新 tag，不重建）。
**安全兜底**：未知/无法判定的路径 → **全量构建**。

### C. 最小重建部署（digest 寻址）

编排器（compose/k8s）**只在镜像引用变化时重建容器**。

- "每次都变的全局 tag" → 全部重建（坏）。
- 改用**不可变 digest（`@sha256`）** → 未变镜像 digest 不变 → 不 recreate。
- 外部/上游镜像**锁到当前运行的 digest**，避免上游 `:latest` 漂移导致有状态服务被动重启。

### D. 合并串联 workflow

多段 `workflow_run` 链（A→B→C）有**调度延迟** + 各自重复的前置 job（如各自的变更检测）。能合并成**单 workflow 内 `needs` 串联**的就合并：前置检测只跑一次，去掉跳转延迟。

### E. 清理

删本地/远端无用分支（如只读、只经发布流前进的 `main` 本地副本）；退役的 workflow 从契约检查**登记为"不可重现"**。

---

## 4. 必避坑（真实踩过，带 why + 检测）

- **合并后别立刻晋升**：发布校验依赖"该 sha 的 CI 已 success"；合并后要**等上游 CI 变绿再晋升**，否则校验直接拒绝。
- **触发模型与 token**：`GITHUB_TOKEN` 推送**不会触发**下游 workflow；受控发布推送要用 **PAT**。
- **buildx 的 digest 不可复现**：即使内容字节相同，每次构建的 image manifest digest 也会变（config 时间戳 / provenance）。**"缓存命中 = digest 相同"是错的** → 最小重建必须靠"不重建未变镜像"（手法 B），不能指望缓存产出相同 digest。
- **跨 workflow 取数据**：不 checkout 仓库时，`gh` 命令必须显式 `--repo`，否则推断不到仓库 → 静默返回空 → 逻辑退化。
- **`bash set -u` + 空数组**：空数组的 `${#arr[@]}` / `${!arr[@]}` 会 `unbound variable` 崩溃（bash 5.2 亦然）；改用普通字符串 + `sort -u`，或用 `${arr[@]+"${arr[@]}"}` 守卫。**只在少见分支触发的路径要专门构造用例测**。
- **静态扫描（如 SonarCloud）**：新 workflow 里的 SSH / 凭据 / `${{ }}`-in-`run` 会被判安全热点/脚本注入。规避：`${{ }}` 只经 `env:` 传值，不直接展开进 `run`；搬迁的旧模式人工 review 后标 safe。

---

## 5. 执行纪律（对 AI 的硬性要求）

1. **一步一 PR**：一个优化一个 PR；本地预检 → CI 闸门 → 合并 → 晋升后用**真实运行日志**验证收益。绝不大爆炸式重构。
2. **默认安全**：任何"只构建/只部署变更"的判定，未知输入一律**退回全量**。over-build 永不发布陈旧代码；under-build 会。**宁可多构建,不可发陈旧。**
3. **契约/文档随设计演进**：改动同步更新契约检查脚本 + 设计文档 + 团队记忆；退役文件登记"禁止重现"，否则制造漂移。
4. **改生产核心路径前确认回退路径**（如保留 fallback 分支）。

---

## 6. 验收标准（实测，非推断）

- 用**优化后运行日志**对比第 2 节基线：CI 运行数 ↓ / 镜像构建数 ↓ / recreate 容器数 ↓ / 端到端耗时 ↓，且**覆盖无缺口**（required check 仍全绿）。
- 抽查一次"无关变更"的发布：未变更的镜像 **digest 不变**、容器**未 recreate**。
- 抽查一次"全量变更"：确认安全兜底生效（全部重建）。

---

## 7. 迁移注记

- 手法 A/D 依赖"分支保护 required check"作为安全网——目标仓库若无,先补。
- digest 寻址（C）需编排器支持按 digest 比较（compose 镜像写 `name@sha256:…`）。
- 本 playbook 的具体脚本/workflow 命名为 vxture 特例,迁移时只取**模式**,按目标仓库技术栈落地。
